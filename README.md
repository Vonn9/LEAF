# Semantic Prompt Search Engine

A semantic search system for prompt discovery that retrieves and ranks the most relevant prompts for arbitrary natural-language queries. Combines bi-encoder vector search, BM25 lexical retrieval, HyDE query expansion, and XGBoost learning-to-rank to surface the most relevant prompts for any natural-language query.

---

## Architecture Overview

Three retrieval methods are implemented and compared side-by-side:

### Baseline — pure vector retrieval
```
Query → bi-encoder embedding → ChromaDB vector search (top-K) → Top-K
```
No reranker and metadata. Establishes the lower bound for both quality and latency.

### Method A — cross-encoder rerank
```
Query → bi-encoder embedding → ChromaDB vector search (top-50) → cross-encoder rerank → Top-K
```
The classical retrieve-then-rerank pattern. Strong quality, but the cross-encoder pass dominates latency.

### Method B — Proposed (HyDE + Hybrid + XGBoost)
```
Query ──► confidence probe (top-3 cosine avg)
               ├─ avg ≥ 0.68 ────────────────────────┐                                   
               └─ avg < 0.68 ──► HyDE (qwen2.5:3b)   │
                            averaged query embedding │
                                      │              │
                                      ▼              ▼
                            ChromaDB vector search (top-50)
                            BM25 lexical search    (top-25)  ◄── union, deduplicated
                                      │
                                      ▼
                            Enrich candidates with 13 features
              (cosine · BM25 · retrieval rank · engagement · category match …)
                                      │
                                      ▼
                               XGBoost ranker → Top-K
```

### Design Decisions

| Choice | Rationale |
|--------|-----------|
| **BGE-base-en-v1.5** bi-encoder | Strong quality/speed balance at 768 dimensions; normalized vectors enable cosine similarity directly in ChromaDB |
| **ChromaDB + HNSW** | Persistent local vector store with approximate nearest-neighbor search — no external service required |
| **BM25 with stemmed bigrams** | Catches exact/stem-match candidates that vector search misses (e.g. "binary search" → bigram `binari_search` prevents false matches on "search" alone) |
| **HyDE** (Hypothetical Document Embeddings) | Local LLM generates a sample prompt that would satisfy the query; averaging its embedding with the query vector bridges the semantic gap for abstract or domain-specific queries |
| **Confidence-gated HyDE** | Top-3 cosine probe decides whether HyDE is needed — precise queries skip the Ollama call entirely, keeping median latency low |
| **Hybrid retrieval** | Union of vector top-50 and BM25 top-25 gives XGBoost a richer, more diverse candidate pool than either method alone |
| **XGBoost Learning-to-Rank** | Combines 13 heterogeneous signals (semantic similarity, lexical match, community engagement, domain alignment) that no single model captures; `rank:pairwise` objective is robust to noisy pseudo-label ordering than direct score regression. |
| **BGE-reranker-v2-m3 as teacher** | Cross-encoder generates offline pseudo-labels per query; paired with the same BGE embedding family for vocabulary consistency |

## Models

| Model | Role | Required? |
|-------|------|-----------|
| `BAAI/bge-base-en-v1.5` | Bi-encoder for document and query embeddings (768-dim, normalized) | **Yes** |
| `BAAI/bge-reranker-v2-m3` | Cross-encoder — Method A reranker + offline teacher for XGBoost pseudo-labels | **Yes** |
| `qwen2.5:3b` via Ollama | Local LLM for HyDE hypothetical document generation | Optional — when Ollama is not available, Method B falls back to the original query embedding |
| `XGBRanker` (`rank:pairwise`) | Learned ranker combining 13 features for Method B | **Yes** (shipped pre-trained) |

## How to Run 😈

All commands are run from the **project root** (the directory that contains `src/` and `dataset.json`). The flow is **Setup → (optional) Explore → Usage → (optional) Advanced**.

### 🛠️ Setup (mandatory, one-time)

These three steps must complete before the demo, evaluator, or any other script will run.

**1. Install Python dependencies**

```bash
pip install -r src/requirements.txt
```

Runtime: ~5–10 min. Python 3.12 recommended.

**2. Pre-download model weights**

Downloads `BAAI/bge-base-en-v1.5` and `BAAI/bge-reranker-v2-m3` upfront with visible HuggingFace progress bars, so later steps don't appear stuck on a silent spinner. Skip only if both models are already in `~/.cache/huggingface/`.

```bash
python src/warmup.py
```

Runtime: 5–30 min on first run (~2.7 GB total). Re-running is a no-op once the cache is populated.

**3. Build the vector index**

Loads `dataset.json`, preprocesses each prompt (title + content + category + subcategory + tags), generates embeddings, and inserts them into ChromaDB. Also precomputes title embeddings for fast `cosine_title` lookup at query time.

```bash
python src/build_index.py
```

Runtime: ~3 min for 20K documents, assuming Step 2 already ran. Output: `chroma_db/`, `title_vecs.npz` at the project root. Both are regenerated locally and never travel inside `src.zip`.

###  Usage (after Setup, run any / all — independent)

#### A. Interactive demo (recommended starting point)

```bash
streamlit run src/streamlit_app.py
```

Four search modes (**Baseline** / **Method A** / **Method B** / **Three-way compare**), live latency badges, a dataset dashboard, and the rendered README. First launch takes ~30 s to load models on top of the Setup cost.

#### B. Reproduce the headline metrics

`evaluate.py` is the single source of truth for the numbers quoted in this README.

```bash
# Quality only (~10 s) — uses xgb_model.json + training_cache_v10.npz from src/
python src/evaluate.py --no-latency

# Full evaluation: quality + end-to-end latency benchmark
python src/evaluate.py
```

The latency portion loads the live `SearchEngine` (cross-encoder + ChromaDB), so it incurs the same warm-up cost as the demo. The `--no-latency` flag skips that entirely.

Expected output (matches the tables further below):

```
Test : 180 queries · P@5 = 0.682 · MRR = 0.900
Method B latency : ~104 ms median (~58× faster than Method A)
```

### 🔧 Advanced — retrain from scratch (optional)

Only needed if you want to verify training reproducibility end-to-end. The shipped artifacts already cover the demo, `evaluate.py`, and every number in this README.

#### Re-train the XGBoost ranker

```bash
python src/XGboost.py
```

Re-uses `src/training_cache_v10.npz` (cross-encoder pseudo-labels over 1200 stratified pseudo-queries × 50 candidates) and re-trains the XGBRanker in seconds. **Without** the cache file this step takes 10–20 min because the cross-encoder must re-score ~60K query–document pairs.

> Bump `CACHE_VERSION` in `XGboost.py` to invalidate the cache when features or queries change.

#### Re-tune hyperparameters with Optuna

```bash
python src/optimization.py
```

50 Optuna trials on the validation split, single evaluation on the held-out test set, best parameters saved to `src/best_params.json` (which `XGboost.py` loads automatically on the next training run). Runtime: ~1 min.

#### Terminal A/B test

Quick sanity-check of the engine without launching Streamlit — runs the built-in 20-query benchmark and prints a three-way comparison table plus latency.

```bash
python src/query.py
```

## Evaluation

Three complementary evaluations are reported. They use different oracles and query sources, so their absolute numbers are not directly comparable — each captures a different dimension of system quality.

### Part 1 — Held-out test set (Method B, quantitative)

180 queries held out from training (pseudo-queries = document titles, never seen during Optuna tuning).
Relevance oracle: normalized cross-encoder label > 0.5 (derived from the same teacher model used to generate training labels).

| Split | P@5 | MRR |
|-------|:---:|:---:|
| Val (180 queries) | 0.743 | 0.940 |
| **Test (180 queries)** | **0.682** | **0.900** |

The validation–test gap (P@5 0.743 → 0.682) is consistent with mild validation-side optimism from Optuna's 50-trial search; with 180 test queries, a gap of this magnitude is also partially attributable to between-split distributional variance. The test number is therefore reported as the primary quality indicator, with the val number disclosed for transparency.

### Part 2 — Latency benchmark (20 queries, top-5, 1 warmup)

This benchmark uses the 20 natural-language queries shipped in `evaluate.py` and `query.py`. It measures the live end-to-end path, including ChromaDB retrieval, BM25, XGBoost inference, and the cross-encoder pass for Method A.

| Method | Median (ms) | P95 (ms) |
|--------|:-----------:|:--------:|
| Baseline — vector only | 26 | 27 |
| Method A — cross-encoder | 6,080 | 8,108 |
| **Method B — Proposed (HyDE + Hybrid + XGBoost)** | **104** | **120** |

Method B is about **58× faster** than Method A on median latency, while still adding BM25, title, category, and engagement features beyond pure vector search. XGBoost inference remains negligible; most of Method B's runtime is retrieval and feature construction.

### Part 3 — Manual natural-language query sanity check

We also evaluated the same 20 manually written natural-language queries by human.

| Method | Human P@5   | Human MRR | 
|--------|:-----------:|:--------:|
|Baseline | 0.810 | 0.925 |
|Method A | 0.890 | 1.000 | 
|Method B | 0.930 | 0.975 | 

The vector-only Baseline already performs reasonably well, but it remains more vulnerable to noisy or only superficially similar matches. Method A improves both metrics, reaching a Human P@5 of 0.890 and the highest Human MRR of 1.000. Method B achieves the best Human P@5 of 0.930 while maintaining a near-perfect Human MRR of 0.975. 
This suggests that the XGBoost+metadata pipeline provides broader top-5 relevance coverage, while sacrificing only a very small amount of rank-1 precision. However, human evaluation still have some limitation subjectively.
